# BUG报告：x31寄存器覆盖导致的死循环

**日期**: 2025-11-27
**状态**: 🔴 **程序BUG（非硬件BUG）**
**严重程度**: 致命 - 导致程序永久死循环

---

## 🔍 问题概述

bubble sort测试程序在运行时陷入永久死循环，寄存器值完全不变，PC一直在ROM196-197之间循环。即使运行10ms，程序也无法完成排序。

### 现象总结

1. ✅ ROM加载正确
2. ✅ 硬件JALR重复跳转已修复
3. ❌ 程序从80us到10ms，寄存器值完全不变
4. ❌ PC永久停留在ROM196-197循环
5. ❌ x31寄存器值错误：**x31 = 0x4310 (ROM196自己的地址！)**

---

## 🐛 根本原因：嵌套函数调用未保存x31寄存器

### 问题机制

在RISC-V架构中，x31 (ra) 寄存器用于存储函数返回地址。但是：
- **`jal x31, offset` 会覆盖x31的旧值**
- **如果函数内部再调用其他函数，旧的返回地址会丢失**

### 错误的调用链

```
ROM184 (0x42e0): jal x31, 12
  ↓ x31 = 0x42e4 (ROM185)，跳转到ROM187

ROM187-196: display_complete函数
  |
  ├─ ROM192 (0x4300): jal x31, 20
  |    ↓ x31 = 0x4304 (ROM193) ← 覆盖了0x42e4！
  |
  |  ROM197-200: delay函数
  |    ROM200: jalr x0, 0(x31) ← 返回到0x4304 (ROM193)
  |
  ├─ ROM193-194: 继续执行
  |
  ├─ ROM195 (0x430c): jal x31, 8
  |    ↓ x31 = 0x4310 (ROM196) ← 再次覆盖！
  |
  |  ROM197-200: delay函数
  |    ROM200: jalr x0, 0(x31) ← 返回到0x4310 (ROM196)
  |
  └─ ROM196 (0x4310): jalr x0, 0(x31)
       ↓ 想返回到ROM184的调用者
       ↓ 但x31 = 0x4310 (自己！)
       ↓ 跳到0x4310 → PC = 0x4310
       ↓ 下个周期PC+4 = 0x4314 (ROM197)
       ↓ 又进入delay函数！
       ↓ 无限循环！
```

### 死循环路径

```
ROM196 (0x4310): jalr x0, 0(x31)  ← x31=0x4310，跳转到自己
                                   ↓ PC=0x4310，下周期PC=0x4314
ROM197 (0x4314): addi x26, x0, 80  ← delay函数入口
ROM198 (0x4318): addi x26, x26, -1 ← 循环体
ROM199 (0x431c): bne x26, x0, -4   ← 跳回ROM198（循环80次）
                                   ↓ x26=0，不跳转
ROM200 (0x4320): jalr x0, 0(x31)   ← 返回到x31=0x4310
                                   ↑
                                   └─ 回到ROM196，继续死循环！
```

## 🔧 正确的函数调用规范

### RISC-V函数调用约定

在有嵌套调用的函数中，必须保存和恢复x31寄存器：

```assembly
# display_complete函数 (正确写法)
display_complete:
    addi sp, sp, -4      # 分配栈空间
    sw   x31, 0(sp)      # 保存返回地址

    # ... 函数体代码 ...

    jal  x31, delay      # 调用delay（会覆盖x31）

    # ... 更多代码 ...

    jal  x31, delay      # 再次调用delay

    # ... 函数结束 ...

    lw   x31, 0(sp)      # 恢复返回地址
    addi sp, sp, 4       # 释放栈空间
    jalr x0, 0(x31)      # 返回
```

### 当前程序的错误

ROM187-196 (display_complete函数) **没有保存x31**：

```assembly
ROM187-191: # 一些初始化代码
ROM192: jal x31, 20    # 调用delay，x31被覆盖
ROM193-194: # 继续执行
ROM195: jal x31, 8     # 再次调用delay，x31再次被覆盖
ROM196: jalr x0, 0(x31) # 返回，但x31已经丢失！
```

**结果**：ROM196返回时，x31不是ROM184设置的0x42e4，而是ROM195设置的0x4310！

## 🎯 解决方案

### 方案1：修改汇编代码（推荐）

在 `display_complete` 函数入口和出口添加x31保存/恢复：

```assembly
display_complete:
    addi sp, sp, -4
    sw   x31, 0(sp)

    # ... 原有代码 ...
    jal x31, delay
    # ... 原有代码 ...
    jal x31, delay

    lw   x31, 0(sp)
    addi sp, sp, 4
    jalr x0, 0(x31)
```

### 方案2：使用不同的寄存器

如果不想使用栈，可以用另一个临时寄存器保存：

```assembly
display_complete:
    add  x29, x31, x0    # 保存x31到x29

    # ... 调用其他函数 ...

    add  x31, x29, x0    # 恢复x31
    jalr x0, 0(x31)
```

**注意**：x29必须是调用者保存寄存器！

### 方案3：使用叶函数优化

如果delay函数不再调用其他函数（叶函数），可以用其他寄存器作为返回地址：

```assembly
ROM192: jal x29, delay   # 用x29代替x31
ROM195: jal x29, delay   # 同样用x29
ROM196: jalr x0, 0(x31)  # 返回到ROM184的调用者
```

## 💡 经验教训

### 教训1：RISC-V函数调用约定至关重要

**ABI规定**：
- **x1-x31 (ra)**: 返回地址寄存器，调用者保存
- **x8-x9 (s0-s1)**: 被调用者保存寄存器
- **x10-x17 (a0-a7)**: 参数寄存器，调用者保存

如果函数内部有嵌套调用，**必须保存ra寄存器**！

### 教训2：叶函数 vs 非叶函数

- **叶函数**：不调用其他函数，可以直接使用x31
- **非叶函数**：调用其他函数，必须保存/恢复x31

`display_complete` 是非叶函数（调用了delay两次），但代码没有保存x31！

### 教训3：调试技巧

当程序陷入死循环时：
1. ✅ 追踪PC执行流程
2. ✅ 检查控制信号（flush, stall, pcSource）
3. ✅ **追踪返回地址寄存器（x31）的值** ← 本次的关键！
4. ✅ 分析函数调用链和嵌套层级
